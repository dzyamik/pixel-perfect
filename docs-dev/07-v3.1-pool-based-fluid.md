# 07 — v3.1 pool-based fluid simulation (planning + design)

> Drafted 2026-05-01 in response to user feedback on v3.0.x:
> "probably there are even more effective approaches, like
> handling group of elements as 1 element."

The v3.0.x series got the mass-based simulation working
(surfaces flatten, no orphans, fast spread, adaptive reach).
Per-cell processing is now efficient — but **fundamentally,
every active fluid cell costs a `stepLiquid` call per tick**.
At 30 000+ active cells the per-cell cost dominates regardless
of how clean the inner loop is.

The proper-scale optimization, used by every production fluid
sim that actually scales (Minecraft's water, DwarfCorp,
Liquid Story Binder, etc.), is to **treat connected pools of
the same fluid as a single object**. v3.1 implements this.

---

## Core idea

Every connected component of same-material fluid cells is a
**pool**:

```typescript
interface FluidPool {
    id: number;
    materialId: number;
    cells: Set<number>;       // flat indices y*W + x
    totalMass: number;        // Σ mass over all cells
    perimeter: Set<number>;   // cells with at least one non-pool 4-neighbor
    surfaceLevel: number;     // y of the top non-full row (computed)
    dirty: boolean;           // shape changed this tick
}
```

Per tick, each pool runs ONE pass of:
1. **Compute target distribution** from `totalMass` and shape.
2. **Apply the distribution** to all cells in one batch.

For a settled pool, step 1 produces no change vs current state
→ no writes → effectively zero cost. Interior cells never touch
the per-cell flow path.

For a dynamically-shaped pool (carve, deposit, drain through
hole), the boundary changes; pool maintenance reshapes it.

---

## What this replaces

Per-cell `stepLiquid` is retained for **single-cell fluid
movement** (a falling drop, a rising gas bubble, a single cell
density-swap into another fluid). It's not gone.

The pool path takes over once cells form a connected component
of size ≥ `POOL_MIN_SIZE` (e.g. 4). Below that, per-cell flow
is fine; above, batch processing wins.

---

## Architecture

### Storage

`ChunkedBitmap` gains a sidecar `_poolIds: Uint16Array | null`.
Each cell stores its pool id (`0xFFFF` = no pool / not a fluid
cell). Pools live in a `Map<number, FluidPool>` on the bitmap.

### Pool maintenance ops

- **Create**: when a fluid cell appears (via `setMass` from
  `air → fluid`), check if any 4-neighbor is in a pool of the
  same material. If yes, join that pool. Otherwise create a new
  singleton pool.
- **Merge**: when joining, if multiple neighbors are in
  different pools (same material), merge. The smaller pool
  joins the larger; cells get re-tagged.
- **Remove**: when a fluid cell disappears (via `setMass` to
  `air` or `setPixel` to a non-fluid material), remove from
  pool. If the cell was a perimeter cell, no further work. If
  it was internal, the remaining pool may have split — defer
  the connectivity check (mark `pool.dirty = true`).
- **Split detection**: at tick start, dirty pools run a
  flood-fill over their cells. If multiple components emerge,
  split into separate pools.

### Per-tick step

1. **Reconcile dirty pools** (flood-fill split detection,
   merge stale references).
2. **For each pool**: compute the equilibrium distribution.
3. **Sub-pool-size cells**: process via the existing
   `stepLiquid` per-cell path.

### Equilibrium distribution

For a pool with `N` cells and `totalMass` units:

- Target mass per cell = `totalMass / N` if every cell can
  hold up to `MAX_MASS + MAX_COMPRESS`.
- For tall stacks: bottom cells over-compress (up to
  `MAX_MASS + MAX_COMPRESS`), upper cells partial-fill.
- The "fill level" `y_level` is the row where mass goes from
  `MAX_MASS` (full) to partial. Cells below `y_level` get
  `MAX_MASS + small compression`. Cells at `y_level` get
  the partial. Cells above are empty (and aren't in the pool).

This is a closed-form per-pool: O(N) to apply.

---

## Phased implementation

**Phase 1 (v3.1.0-alpha)** — pool detection only. Run flood
fill at tick start; build pools; print stats. Verify membership
matches per-cell behavior. No behavior change.

**Phase 2 (v3.1.0-beta)** — pool-level equilibrium distribution.
For settled pools (no boundary changes), apply the distribution
without per-cell flow. Cells outside pools (singletons, falling
drops) keep using `stepLiquid`.

**Phase 3 (v3.1.0)** — incremental pool maintenance. Replace
per-tick flood fill with merge / split / remove ops triggered
by `setMass` calls. Pools are maintained in O(1) for most
operations; flood fill only when split detection is needed.

---

## Trade-offs

Pros:
- Settled pools cost O(1) (just a "no change" check).
- Dynamic pools cost O(N) for the redistribution pass, vs
  O(N × ~12 transfers) for per-cell flow.
- Surface level is exact, no per-cell mass jitter.

Cons:
- Complexity: pool maintenance is fiddly (split detection
  edge cases, cross-material density swaps for boundary
  cells, fire/sand still per-cell).
- Memory: per-cell pool id (`Uint16Array` = +2 bytes / cell)
  + the pool registry.
- Correctness risk: pool state must stay in sync with cell-
  level state. Bugs here produce mass non-conservation or
  visible artifacts.

---

## What stays the same

- Sand / fire / static — unchanged. They don't go through
  pools.
- Cross-material density swaps — still atomic per-cell. The
  pool's perimeter cells enter the per-cell path when they
  contact a different-material cell.
- Active-cell tracking — pool dirty flags replace some of the
  per-cell active set tracking, but cells outside pools still
  use it.
- Renderer — unchanged. Pool state is a sim-side
  optimization, not a rendering change.

---

## Open questions

1. **Cross-chunk pools**: a pool can span many chunks. Pool
   storage is per-bitmap, not per-chunk. Performance
   implications for very large worlds?
2. **Fire pools**: should burning materials within fire form
   a pool? Probably not — fire is per-cell already.
3. **Gas pools**: gas rises; the "fill level" is at the TOP
   of the pool (against ceilings). Inverted formula. Same
   underlying mechanism.
4. **Pool membership at chunk boundaries**: when a chunk is
   carved or deposited (modifying many cells), how to
   incrementally update pool memberships without a full flood
   fill?

## Estimated scope

- Phase 1: ~1 day. Pool detection + tracking, no behavior
  change.
- Phase 2: ~1-2 days. Equilibrium distribution + integration.
- Phase 3: ~2 days. Incremental maintenance + edge cases.

Total: ~4-5 days of focused work for a stable v3.1.

If v3.0.4's adaptive reach is "good enough" in practice, v3.1
becomes lower priority. Worth measuring (via the new demo-09
